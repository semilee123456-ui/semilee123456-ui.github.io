그래서 작업 방식은 항상 이렇습니다:
1. Claude가 로컬에서 파일을 고치고 로컬 git에는 커밋함
2. `preview.html`(styles.css/script.js를 index.html에 인라인으로 합친 자체완결 파일)을 만들어 Artifact로 게시 — 이번 세션 고정 URL: `https://claude.ai/code/artifact/f3fa44f5-e368-4da5-b436-fc3684207fc3` (같은 URL에 재게시)
3. 바뀐 파일 전체 zip과 git bundle을 `SendUserFile`로 사용자에게 전달
4. **사용자가 GitHub 웹사이트에서 직접 업로드**해야 실제 사이트에 반영됨 — 파일이 100개 넘으면 GitHub 웹 업로드가 "Yowza, that's a lot of files" 에러를 내므로, **전체 zip 대신 `git diff --name-only <이전 브랜치 시작점> HEAD`로 실제 바뀐 파일만 추려서 zip으로 드릴 것**

**이 문서를 쓰는 시점 기준, 이 브랜치(`claude/chamtax-handover-2mxrb6`)의 커밋은 로컬에만 있고 GitHub에 반영되지 않았을 수 있습니다.** 새 세션을 시작하기 전에 사용자가 zip/bundle을 실제로 업로드했는지 먼저 확인하세요 — 안 했다면 새 세션(컨테이너가 새로 시작되며 GitHub에서 다시 clone됨)은 오늘 작업한 내용을 전혀 모르는 상태로 시작합니다. **실제로 이번 세션 중간에 이 문제가 한 번 발생했습니다** — 이전 세션의 로컬 커밋이 컨테이너 초기화로 통째로 날아갔고, `git log`로 확인해보니 origin/main에는 사용자가 그동안 업로드해온 "Add files via upload" 커밋들만 남아있었습니다. 작업을 시작하기 전에 항상 `git log --oneline -10`으로 지금 보이는 커밋들이 정말 최신인지 의심해보세요.

- `POWERBALL_DRAW_ARCHIVE`(1,378건, 2015-10-07~2026-07-20), `MEGAMILLIONS_DRAW_ARCHIVE`(910건, 2017-10-31~2026-07-17): 당첨번호만 있는 아카이브. "내 번호 vs 역대 당첨번호" 비교 위젯의 비교 대상을 7건→2,289건으로 확대. **번호가 많아서 위젯은 역대급 5건 + 베스트매치(점수순) 5건만 카드로 보여주고, 전체 비교 건수는 안내 문구에만 표시**하는 방식으로 설계함(2000개+ 카드를 다 그리면 페이지가 무한정 길어지는 문제 방지)
- `MEGAMILLIONS_JACKPOT_ARCHIVE`(832건, 2018-02-02~2026-01-20), `POWERBALL_JACKPOT_ARCHIVE`(182건, 2025-05-24~2026-07-20): **잭팟 금액까지 있는** 아카이브(당첨번호 아카이브와 별개 — 당첨번호 CSV엔 금액이 없어서 사용자가 usamega.com의 "Past Results" 페이지를 별도로 복사해서 제공함). "역대 잭팟 확인 기록" 패널의 비교 대상을 9건→1,021건으로 확대, 여기도 "더 보기" 버튼으로 페이지네이션(`JACKPOT_HISTORY_PAGE_SIZE=15`, `expandJackpotHistory()`)
- **파워볼 데이터 파싱 시 주의**: usamega.com Powerball Past Results엔 "Double Play"라는 별도 게임 번호 블록이 껴있어서(날짜 다음에 번호6개+배수, 그다음 "Double Play"+빈줄+번호6개, 그다음에야 잭팟 금액) 메가밀리언즈용 파서를 그대로 못 씀 — `findDrawByDate()`/파싱 스크립트에서 "Double Play" 다음 6줄을 건너뛰는 처리가 되어 있음
- "자주 나온 번호" 문구: 예전엔 하드코딩된 고정 텍스트였는데, 이제 `computeNumberFrequency()`가 위 당첨번호 아카이브에서 매번 실제로 계산함(`renderNumberFrequencyStats()`)

**신규 기능**
- 🏆 **실수령액 기준 재랭킹**(`getJackpotRealTakeHomeRanking()`/`renderJackpotTakeHomeRanking()`): 발표된 잭팟(연금) 순위가 아니라, 한국 거주자가 **오늘 세율로 실제 받는 금액** 기준으로 실제 잭팟 기록 1,000건+를 재정렬해 TOP 10을 보여줌. "발표 1위가 실수령액 1위가 아닐 수 있다"가 포인트. 미국 세율만 다루는 다른 복권 통계 사이트(USA Mega 등)는 외국 거주자 세금 계산 자체를 안 하니 만들 수 없는 조합이라 판단해서 만듦
- 📅 **날짜로 당첨번호 찾아보기**(`findDrawByDate()`/`renderDateLookupResult()`): 날짜 하나를 고르면 그날 당첨번호+잭팟금액(있으면)을 바로 보여줌. 목록형 검색(USA Mega의 Search Past Results처럼 여러 회차를 한번에)이 아니라 단일 날짜 결과만 보여주는 쪽으로 사용자와 상의해서 결정함 — 조회 우선순위는 `JACKPOT_HISTORY`(큐레이션) → `*_JACKPOT_ARCHIVE`(금액 있음) → `*_DRAW_ARCHIVE`(번호만, 범위 더 넓음)

**다국어**
- `LANGUAGE_CONTENT_PAGES`(26개국 SEO 랜딩페이지 링크 목록) 중 7개 항목(아랍어권·프랑스어권·스페인어권·포르투갈어권·대만홍콩·동티모르·우크라이나)에 17개 언어 `more` 객체가 아예 빠져있어서, 그 17개 언어로 보는 사용자에게 스크린리더용 aria-label이 영어로 폴백되고 있었음(화면 텍스트 자체는 정상이었음) — 사용자가 제미나이+GPT 둘 다에게 번역 받아서 상호 대조 후 반영함(두 결과가 의미상 일치해서 신뢰도 확보). 정확도 우려되는 희소 언어(크메르어·라오어·싱할라어·키르기스어 등)일수록 이렇게 **2개 이상 AI로 교차검증** 하는 게 유용했음
- `odds.howtoText`(파워볼 설명)는 확인해보니 이미 22개 언어 전부 번역되어 있었음 — 예전 인수인계 문서(`handoff-2026-07-21.md`)에 "6개 언어만 지원"이라고 남아있던 게 stale한 정보였음. **예전 인수인계 문서의 "남은 이슈" 목록을 맹신하지 말고, 실제 코드를 다시 확인하고 시작할 것**

## 3. 코드 아키텍처 핵심 패턴

- **다국어**: JSON 기반(`data-i18n*`, `i18n-source/translations.json` → `node scripts/build-i18n.js` → `i18n/*.json`)과 JS 인라인(`pickLang(ko,en,zh,vi,th,ru,more)`) 두 시스템이 공존. **정적 HTML 텍스트는 JSON, 계산 결과/동적 조합 텍스트는 pickLang**이 일반적인 구분 기준. `more` 인자 빠뜨리면 17개 언어가 조용히 영어로 폴백되는 게 이 코드베이스에서 가장 흔한 버그 유형
- **거대 데이터 배열들**: `POWERBALL_DRAW_ARCHIVE`/`MEGAMILLIONS_DRAW_ARCHIVE`/`POWERBALL_JACKPOT_ARCHIVE`/`MEGAMILLIONS_JACKPOT_ARCHIVE`는 전부 `[date, [numbers], special, (amountMillions)]` 형태의 압축 튜플 배열로, 한 줄짜리 매우 긴 `const` 문. **Read 도구로 이 줄 주변을 읽으면 토큰 초과 에러가 남** — 대신 `awk '/패턴/{print NR; exit}'`로 줄 번호만 찾거나, Python으로 정규식 치환해서 통째로 교체하는 방식을 쓸 것 (`re.sub`으로 `const XXX = \[.*?\];` 패턴 찾아 바꾸기)
- **잭팟 데이터 병합**: `getCombinedJackpotHistory()`가 `JACKPOT_HISTORY`(수동 큐레이션, notes 등 있음) + 두 `*_JACKPOT_ARCHIVE`를 합쳐서 반환. 여러 곳(잭팟 기록 패널, 실수령액 랭킹, 날짜 검색)에서 재사용됨. 항목 수가 많아서 **다 DOM에 그리지 말고 페이지네이션하거나 top-N만 추릴 것** — 이미 두 군데(번호 비교 위젯, 잭팟 기록 패널)에서 이 실수를 할 뻔했다가 고쳤음
- **`calcTakeHome(amount, country, stateCode)`**: amount는 "억원" 단위, 반환값의 `final`이 세후 실수령액(억원). 새로운 "실제 금액 계산" 기능을 만들 때 이 함수를 그대로 재사용
- **`COUNTRY_TAX_PROFILES`**: 20개국 세금 프로필. 새 나라 추가 시 여기 + `calcTakeHome()` + 국가 지도 좌표까지 3곳 갱신 필요
- **`formatWon(n)`**: 원화 금액을 언어별로 표시(억/조 vs million/billion)하는 핵심 함수

## 5. 자체완결 프리뷰 만드는 법

import re
html = open('index.html', encoding='utf-8').read()
css = open('styles.css', encoding='utf-8').read()
js = open('script.js', encoding='utf-8').read()
html2 = re.sub(r'<link rel="stylesheet" href="styles\.css[^"]*">', lambda m: '<style>\n' + css + '\n</style>', html)

open('preview.html', 'w', encoding='utf-8').write(html2)

- **파워볼 잭팟 금액 데이터가 182건뿐**(메가밀리언즈 832건보다 훨씬 적음) — 사용자가 usamega.com Powerball Past Results를 아직 최근 14개월치만 복사해줌. 더 받으면 `POWERBALL_JACKPOT_ARCHIVE`에 계속 합칠 수 있음 (파싱 스크립트는 위 2번 섹션 참고, "Double Play" 블록 스킵 필요)
- **잭팟 금액 아카이브가 완전한 전체 역사는 아님**: 메가밀리언즈는 2018년부터, 파워볼은 2025년부터만 금액 데이터가 있음. 당첨번호 아카이브(금액 없음)는 이보다 훨씬 예전(2015/2017)부터 있음 — `findDrawByDate()`가 이 차이를 자동으로 처리해서, 금액 없는 오래된 날짜는 "번호만 있음" 안내를 보여줌
- **언어 혼용 감사**: 이번 세션 초반에 서브에이전트를 백그라운드로 띄웠는데 끝까지 응답이 없었음(정상 종료 안 된 것으로 추정). 사용자와 상의 후 포기하기로 함 — 재시도한다면 백그라운드에 맡기지 말고 직접 몇 개 언어만 빠르게 확인하는 쪽을 추천
- **국가 지도 실제 지리 정보화**: 사용자가 "지금 단순 도형을 더 알아보기 쉽게만" 원해서 라벨만 추가함(위 2번 참고). 실제 국경선 기반으로 바꾸려면 이 환경엔 없는 GeoJSON 데이터가 필요 — 네트워크가 열리는 환경이거나 사용자가 SVG 파일을 직접 주면 가능
- **국가별 실수령액 재랭킹 기능은 한국 거주자 기준으로 고정**되어 있음(국가 선택 드롭다운 없음) — 필요하면 확장 가능하나, 지금은 단순하게 유지함
