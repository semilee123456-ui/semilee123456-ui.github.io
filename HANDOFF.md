## ⚠️ GitHub push가 항상 막혀있음 (구조적 제약, 매 세션 재확인 불필요)

이 세션(Claude Code)은 **GitHub에 직접 push할 권한이 없습니다** — `git push` 시도하면 항상
`403`(Anthropic GitHub App이 Authorize만 되고 Install은 안 된 상태로 이미 진단 완료된 알려진
이슈). GitHub API(`push_files`, `create_pull_request`)도 마찬가지로 막혀있음을 확인했습니다.
재진단하지 말고 아래 방식을 그대로 따르세요:

1. Claude가 로컬에서 파일을 고치고 로컬 git에는 커밋함
2. `SendUserFile`로 바뀐 파일(들)을 사용자에게 직접 전달 — 여러 개면 zip 하나로 묶어서 전달
3. **사용자가 GitHub 웹 UI에서 "파일 업로드"로 직접 덮어씀 (파일을 지워야 할 때는 사용자가
   GitHub 웹 UI에서 직접 삭제 — zip 업로드만으로는 삭제가 안 됨, 위 섹션 참고)**

커밋 서명 관련 훅 경고("Unverified")는 매번 뜨는데, 실제로는 커밋에 SSH 서명(`gpgsig`)이
정상적으로 들어가 있고, 단지 로컬 검증에 쓰이는 서명 공개키 파일이 비어있어서 로컬에서만
"N"(미검증)으로 표시되는 환경 설정 문제입니다. 이 브랜치를 push하지 않는 워크플로우라 실제
GitHub 표시에는 영향 없음. 해결 불가능/불필요한 표시상의 문제이니 매번 재진단하지 마세요.

**파일명 하이픈 버그**: 사용자가 GitHub 웹 "Upload files" 화면에 하이픈(`-`)이 든 파일을
드래그해서 올리면, **업로드 결과물에서 하이픈이 사라진 채로 저장소에 반영됨**(예:
`odds-data.js`를 올렸는데 실제로는 `oddsdata.js`라는 별개 파일로 커밋됨). 이미 저장소에
같은 이름(하이픈 포함)의 파일이 있어도 그 파일을 덮어쓰는 게 아니라, 하이픈 없는 새 파일이
따로 생겨버림 — 즉 "덮어쓰기"를 의도했어도 실제로는 업데이트가 반영되지 않고 구버전 파일이
그대로 라이브에 남는다는 뜻. **2026-07-23 세션에서 실제 사고로 확인됨**: `odds-data.js`
갱신본이 `oddsdata.js`로 잘못 올라가 있었고, 정작 라이브의 `odds-data.js`는 구버전(172KB,
최신은 354KB)인 채로 몇 주간 방치되어 사이트가 옛날 확률/잭팟 데이터를 계속 보여주고 있었음.
**대응**: (1) 새 파일을 만들 때는 하이픈 대신 언더스코어(`_`)를 기본으로 쓸 것. (2) 하이픈이
든 기존 파일(`odds-data.js` 등)을 업데이트해야 할 때는, 업로드 후 반드시 저장소 파일 목록에서
하이픈 없는 이름의 파일이 새로 생기지 않았는지, 그리고 원래 파일의 커밋 시각/크기가 실제로
바뀌었는지 확인할 것 — "업로드했다"만으로 반영됐다고 가정하지 말 것. 이미 라이브에 있는
기존 파일명(하이픈 포함)은 검색엔진 인덱싱 때문에 이름 자체를 바꾸지 말 것.

### 다국어(i18n) 작업 방법
1. `i18n-source/translations.json`에서 해당 키의 번역을 수정 (**`i18n/*.json`을 직접 고치지
   말 것** — 다음 빌드 때 덮어써짐). 큰 JSON 파일 전체를 `json.dump()`로 재직렬화하면 기존의
   수작업 포맷팅이 깨져서 거대한 diff가 남으므로, 바뀌는 부분만 타겟팅한 문자열 치환으로 수정할 것.
2. `node scripts/build-i18n.js` 실행 → `i18n/*.json` 22개 재생성
3. HTML 정적 텍스트: `data-i18n`/`data-i18n-html`/`data-i18n-placeholder`/`data-i18n-aria-label`
   속성 + `resolveI18n(key)`가 `i18n/{lang}.json`을 fetch해서 치환. **주의**: plain `data-i18n`은
   언어 전환마다 `el.textContent`로 통째로 덮어써지므로, 원문 안에 `<span>` 같은 마크업을 넣어도
   다음 전환 때 사라짐 — 마크업이 필요하면 `data-i18n-html`을 쓸 것(`tests/i18n_attr_lint.js`가
   이 오사용을 사전 경고함).
4. JS 동적 문자열: `pickLang(ko, en, zh, vi, th, ru, more)` — 6개는 위치 인자, 나머지 17개는
   `more` 객체. **`more`를 빠뜨리면 그 17개 언어에서 영어로 조용히 폴백됨** — 과거 세션에서
   발견된 버그의 상당수가 이 실수였음.
5. **새 `data-i18n*` 키를 추가했으면 반드시 `tests/i18n_coverage_audit.js`를 돌려서 22개 언어
   전부에 번역이 채워졌는지 확인할 것.** 2026-07-23에 FTC 체크리스트 관련 키 10개가 번역
   누락 상태로 몇 세션째 라이브에 떠 있던 걸 이 테스트를 처음 만들면서 발견함(21개 외국어에서
   그 부분만 한국어 원문이 그대로 노출되고 있었음) — 새 기능 추가 시 "22개 언어 번역함"이라고
   적어놓고 실제로는 빠뜨리는 실수가 반복되니, 서술이 아니라 이 테스트 통과 여부로 확인할 것.

## 코드 아키텍처 핵심 패턴

- **`sharedCountry`**: 'kr'/'us'/'cn'/'in'/... — 사용자가 고른 "세금 계산 기준". 홈 화면과
  국가비교 화면이 이 값을 공유함. `data-basis="kr"` 같은 속성으로 FAQ 항목을 세금 기준별로도
  필터링함(`filterFaq()`).
- **`currentLang`**: 'ko'/'en'/... — 표시 언어. localStorage에 저장 안 되고 새로고침하면
  'ko'로 리셋됨(`?lang=` URL 파라미터로 시작은 가능, 적용 후 URL에서 자동 제거).
- **FAQ 대상 배지**: `data-basis`가 "어느 나라 세법을 적용받는지"만 구분하고 "당신이
  누구인지"(재외국민 vs 한국 거주 외국인)는 구분 못 함 — 특정 페르소나 전용 질문에는
  `faq-audience-badge` 클래스로 별도 배지를 붙임(예: "🌏 해외 거주 한국인 전용").
- **다크모드**: `:root[data-theme="dark"]`, `localStorage.getItem('theme')`,
  `toggleTheme()`/`syncThemeToggleIcon()`(script.js). `--teal`/`--status-red`/`--gold`/`--navy`는
  "다크모드에서 글자색으로 쓰일 때" 기준으로 튜닝되어 있어서, 이 색을 **배경**으로 쓰고 흰
  글자를 얹는 배지/버튼에는 하드코딩된 `color:#fff` 대신 `color:var(--on-accent)`를 쓸 것
  (라이트=흰색, 다크=잉크색으로 자동 전환됨).
- **접근성 폰트 floor**: `--fs-small:16px`가 명시적 최소값(시니어 가독성 연구 근거 주석 있음).
  랜딩페이지들은 자체 CSS 변수를 따로 가지고 있어 메인 앱과 별도로 확인 필요.
- **`COUNTRY_TAX_PROFILES`**: 국가별 세금 프로필. 새 국가 추가 시 여기 + `calcTakeHome()` 세율
  로직 + 국가 지도 좌표(`COUNTRY_MAP_COORDS`/SVG path, flagCode로 매칭) 3곳을 같이 갱신.
  국가 지도(`#countryMapWrap`)는 **"국가별 비교" 탭(view-compare)에만 있음**(홈 탭 아님 —
  테스트 짤 때 실수하기 쉬움, 아래 "테스트 짤 때 흔한 함정" 참고).
- **`formatWon(n)`**: 원화 금액을 언어별로 다르게 표시(억/조 vs million/billion). 비슷한 요구가
  생기면 이 함수의 분기 패턴을 참고.
- **`highlightCountryOnMap()`**(script.js): 국가별 비교 탭에서 카드를 탭하면 지도가 스크롤 밖에
  있을 때만 `scrollIntoView({block:'nearest'})`로 보이게 함. 이 스크롤 로직을 건드릴 일이 있으면
  `tests/map_scroll_audit.js`로 회귀 확인할 것.
- **비파괴적 non-breaking space 수정 패턴**: 한국어는 "단어(괄호)"처럼 공백 없이 붙여 쓰는
  경우가 많은데, 좁은 화면에서 이 지점이 그대로 줄바꿈되면 문장이 잘린 것처럼 보임. 고치는
  방법 2가지:
  - 원래 공백이 있어야 자연스러운 자리(예: 라벨과 괄호 사이)는 일반 공백을 NBSP(U+00A0,
    파이썬에서 `'\xa0'`로 명시적으로 써야 함 — 터미널에 직접 타이핑하면 전송 과정에서 일반
    공백으로 뭉개지는 경우가 있었음)로 바꿔서 그 지점에서 줄바꿈이 안 일어나게 함.
  - 원래도 공백이 없어야 자연스러운 자리(예: "주(State)에서"의 ")"와 "에서" 사이)는 NBSP로도
    막히지 않음(CJK 스크립트는 거의 모든 글자 경계에서 줄바꿈이 허용되기 때문) —
    `<span style="white-space:nowrap">...</span>`로 통째로 감싸야 함. 이 경우 `.textContent`가
    아니라 `.innerHTML`로 대입해야 마크업이 살아남음.
  - **주의(2026-07-23 발견)**: `display:flex; justify-content:space-between`인 컨테이너
    (예: `.calc-detail-summary`) 안에 이 nowrap `<span>`을 넣으면, 원래 텍스트가 "텍스트노드 +
    span + 텍스트노드" 3개의 flex item으로 쪼개져서 space-between이 그 사이 간격을 화면
    전체로 벌려버리는 별개의 버그가 생김(줄바꿈 문제가 아니라 flex 레이아웃 문제). 이런
    컨테이너에서는 nowrap span을 추가할 때 전체 텍스트를 감싸는 바깥 `<span>` 하나를 더 씌워서
    flex item 개수를 1개로 유지할 것.

## 테스트 짤 때 흔한 함정 (Playwright로 감사 스크립트 짤 때 참고)

1. **`getComputedStyle(el).display === 'none'`만으로는 조상 요소의 `display:none`을 못 잡음.**
   `display`는 상속되는 속성이 아니라서, 부모가 `display:none`이어도 자식 요소 자신의 계산된
   `display` 값은 그대로 `block` 등으로 나옴. **실제 렌더링 여부는
   `el.getClientRects().length === 0`로 확인할 것** — 조상 체인 어디서 `display:none`이 걸려도
   정확히 잡힘.
2. **어느 탭(view)에 있는지 먼저 확인할 것.** 예: 국가 지도는 `view-compare`에 있지 홈
   (`view-home`)에 없음. 엉뚱한 탭에서 요소를 찾으면 애초에 화면에 없어서
   `getBoundingClientRect()`가 전부 0을 반환하고, "이슈 없음"이 사실은 "아무것도 안 본" 거짓
   통과가 됨.
3. **0건이 나왔다고 끝내지 말고, 일부러 버그를 재현시켜서 탐지되는지 네거티브 컨트롤로
   확인할 것.** (코드를 임시로 한 줄 주석 처리 → 테스트 재실행 → 실패하는지 확인 → 원복). 이
   원칙 자체로 2026-07-23에 진짜 버그(FTC 번역 누락)를 하나 찾아냈음 — 테스트를 새로 만드는
   행위 자체가 버그 탐지로 이어진 사례.
4. **NBSP(U+00A0) 같은 non-ASCII 문자를 Python/JS 소스에 직접 타이핑하지 말 것.** 도구 호출을
   거치는 과정에서 일반 공백으로 뭉개지는 게 간헐적으로 관찰됨 — `'\xa0'`처럼 ASCII escape
   시퀀스로 명시할 것(파이썬은 소스 코드 안에서 이 시퀀스를 안정적으로 해석함).
5. **파일에 쓰는 문자열 치환은 `expected_count`를 지정해서 검증할 것.** 같은 문구가 눈에
   보이는 UI 텍스트뿐 아니라 `<script type="application/ld+json">` SEO 구조화 데이터 블록에도
   중복되어 있는 경우가 많아서(FAQ 스키마 등), 무심코 전체 치환하면 JSON-LD를 깨뜨림. 개수가
   안 맞으면 더 구체적인 앞뒤 문맥으로 앵커를 좁힐 것.
6. **Playwright 브라우저 컨텍스트의 기본 언어는 'ko'가 아닐 수 있음.** locale 설정 없이
   그냥 `page.goto()`만 하면 사이트가 영어로 뜨는 경우가 있었음 — 한국어 기준으로 테스트하려면
   `newPage({ locale: 'ko-KR' })` + `?lang=ko` 둘 다 명시할 것.
7. **이 실행 환경은 외부 인터넷이 막혀있어서(폰트 CDN·광고·환율 API 등) "Failed to load
   resource" 콘솔 에러가 항상 남.** `console_error_audit.js`는 이런 외부 리소스 실패를
   필터링하고 로컬(127.0.0.1) 리소스 실패나 진짜 JS 예외만 잡도록 만들어져 있음 — 비슷한
   테스트를 새로 만들 때 이 필터링을 빠뜨리면 매번 40개 안팎의 가짜 이슈가 뜬다.
